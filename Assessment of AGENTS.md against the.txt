4b. Table Row — View Mode
Matches exactly
- Edit and Delete buttons appear in the actions column in default state.

Differences
- UI-PATTERNS.md view mode row is text-only with hover:bg-gray-50.
- Covenants and CAS Findings always render inputs/select/textarea but disabled when not editing; no hover:bg-... on rows. Example (Covenants):
  - Category cell: <Select ... disabled={!r.editing}>...</Select>
  - Covenant Type: <input ... disabled={!r.editing} />
  - Order: <input type="number" ... disabled={!r.editing} />
- Primary key columns remain text-only:

4c. Table Row — Edit Mode
Matches exactly
- PK field remains plain text during edit.

Differences
- UI-PATTERNS.md calls for swapping to a distinct edit row with bg-blue-50 and Save/Cancel replacing the Edit/Delete buttons.
- Actual:
  - No bg-blue-50; same row style; only disabled toggles to enable inputs
  - Buttons: three buttons always exist; first toggles between “Edit” and “Cancel”; Save remains visible but disabled when not editing or not dirty:
    - <Button size="xs" variant="outline"> {r.editing ? "Cancel" : "Edit"} </Button>
    - <Button size="xs" variant="outline" disabled={!r.editing || !r.dirty || r.saving}> {r.saving ? "Saving…" : "Save"} </Button>
    - <Button size="xs" variant="danger" disabled={r.saving || r.deleting}> {r.deleting ? "Deleting…" : "Delete"} </Button>

4d. Delete Confirmation Mode
Matches exactly
- Delete action requires a confirmation step before executing.

Differences
- UI-PATTERNS.md uses an inline row transform with bg-red-50 and “Are you sure?” text + Confirm/Cancel inline buttons.
- Actual uses a modal dialog overlay with a11y and focus trap:
  - Wrapper: <div className="fixed inset-0 z-[60] flex items-center justify-center">
  - Backdrop: <div className="absolute inset-0 bg-black/40" onClick={() => setConfirmOpen(false)} />
  - Dialog: role="dialog" aria-modal="true" aria-labelledby="confirm-title" aria-describedby="confirm-desc", focus management and Escape handling in useEffect
  - Buttons in modal footer; no inline row changes

Final notes
- Functionally, the pages implement filtering, create, edit, delete, and feedback as intended. However, they diverge from UI-PATTERNS.md in class naming (slate/emerald vs gray/blue/green), markup structure (grid vs flex; no shell wrappers), composition (custom Select/Button, modal instead of inline confirmation), and state handling (toast array vs single successMessage; disabled-toggled inputs vs swapped view/edit rows).
- If strict adherence to UI-PATTERNS.md is required, refactors would include: adopting the page shell wrappers, switching to native controls with the specified class sets or updating patterns to bless the shared components, moving toast to bottom-right with the green bordered style and single successMessage state, moving delete confirmation inline, adding edit-row bg-blue-50, and reworking loading to use a spinner instead of skeletons—or updating UI-PATTERNS.md to reflect the current implemented design system (shared Button/Select, modal confirmation, skeleton loading, top-right toasts, slate palette).
