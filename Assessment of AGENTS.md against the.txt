Assessment of AGENTS.md against the actual repository

Verdict: Partially aligned at a high level (Next.js frontend + .NET backend in a monorepo), but multiple sections are out of sync with the codebase and conventions actually used. Below is an evidence-based comparison and concrete improvements.

What matches

- Monorepo and folders
  - AGENTS lists: frontend/, backend/, docs/, discovery/, pipelines/, scripts/, tools/, legacy/. All are present at the repo root.

- Frontend platform and config
  - Next.js App Router present: frontend/src/app/ exists; many pages under app/.
  - TypeScript and Tailwind present: frontend/tsconfig.json, tailwind.config.ts, postcss.config.js, next.config.mjs exist.
  - Common frontend structure folders exist: frontend/src/components, hooks, models, services, lib, auth.

- Backend solution and layering (at a high level)
  - Solution file exists: casrr.sln.
  - Projects exist for API, Application, Domain, Infrastructure under backend/src.
  - Controllers exist and use [Authorize] policies in at least some controllers (example below).

Key differences and inaccuracies (with evidence)

1) Authentication model (document says Windows/AD without JWT; code uses AAD JWT + policies and Graph)
   - AGENTS.md: “Authentication — Windows / AD Authentication; [Authorize] on all controllers. No JWT, no cookies.”
   - Actual:
     - Program wiring uses JWT bearer/AAD via Microsoft.Identity.Web:
       - backend/src/Casrr.Api/Extensions/StartupExtensions.cs → AddAuthenticationConfigured uses AddMicrosoftIdentityWebApi or AddJwtBearer.
     - Swagger includes “Bearer” and Graph token schemes.
     - Controllers require policies:
       - Example: backend/src/Casrr.Api/Controllers/CovenantsController.cs has [Authorize(Policy = "RequireActiveUser")].
     - Graph dependencies present:
       - backend/src/Casrr.Api/Auth/Graph/*, plus registrations in Program/Startup.
   - Conclusion: Update doc to reflect Azure AD JWT bearer with authorization policies and optional Graph usage; remove “No JWT”.

2) Data access/ORM (document mandates EF Core-only; code uses raw ADO.NET and likely Dapper)
   - AGENTS.md: “ORM — Entity Framework Core. No Dapper, no raw ADO.NET, no stored procedures. Lives in Infrastructure.”
   - Actual:
     - Repositories use SqlClient with SQL strings:
       - backend/src/Casrr.Infrastructure/SqlServer/SqlCovenantsRepository.cs uses SqlConnection/SqlCommand extensively (raw ADO.NET).
     - Dapper.dll is in API bin output (Casrr.Api/bin/Debug/net8.0/Dapper.dll), indicating Dapper is referenced somewhere in the solution.
     - An EF DbContext is registered (AddDbContextPool<CasrrDbContext>), but repository implementations found are ADO.NET-based.
   - Conclusion: Either the code should be refactored to the EF Core pattern in AGENTS.md, or AGENTS.md must be updated to reflect current ADO.NET/Dapper repository approach.

3) Backend project count and structure (document says “7 projects”; repo has 4 core + tests)
   - AGENTS.md: “Solution — 7 projects.”
   - Actual:
     - backend/src contains: Casrr.Api, Casrr.Application, Casrr.Domain, Casrr.Infrastructure.
     - Tests exist under backend/Unit Tests/… but not counted as part of “7”.
   - Conclusion: Replace “7 projects” with an exact current list of projects (4 core + list the test projects explicitly if needed).

4) API route standards (document suggests versionless /api/{resource}; code uses versioned /api/v1)
   - AGENTS.md: Endpoint examples without version segment (e.g., /api/loan-codes).
   - Actual:
     - Example: CovenantsController route is [Route("api/v1/covenants")].
   - Conclusion: Update doc to standardize on versioned routes (/api/v1/{resource}) if that is the current convention, or adjust controllers.

5) Backend layering details and folder conventions
   - AGENTS.md expects:
     - Application/Interfaces/I<Entity>Repository.cs
     - Infrastructure/Persistence/Configurations/<Entity>Configuration.cs with EF
     - Domain entities with no EF attributes
   - Actual:
     - Application interface files live at Application root (e.g., backend/src/Casrr.Application/ICovenantsRepository.cs), not under Interfaces/.
     - Infrastructure repositories live under Infrastructure/SqlServer/* with raw SQL.
     - No EF IEntityTypeConfiguration files were seen in the listing; DbContext class exists as Backend_TemplateDbContext.cs but the implementation pattern is not the one mandated.
   - Conclusion: Either move interfaces into an Interfaces/ folder and implement EF configuration as documented, or update AGENTS.md to reflect the current ADO.NET repository layout under Infrastructure/SqlServer.

6) Frontend service/hook and file-location conventions
   - AGENTS.md expects:
     - Services in frontend/src/services/ with Service suffix (e.g., loanCodesService.ts).
     - All types/interfaces under frontend/src/models/ (no local types in pages/components).
     - Page-specific components under components/<feature>/ subfolder.
   - Actual:
     - Services live under frontend/src/services/api/ and are named by resource (e.g., covenants.ts, samples.ts) without “Service” suffix.
     - Local types file present under a route: frontend/src/app/admin/users/types.ts (contradicts “all models under src/models”).
     - Many page-specific components are co-located under route folders:
       - Example: frontend/src/app/review/[ecif]/review-info/components/* are inside the app route, not under src/components/<feature>/.
   - Conclusion:
     - Either move local types into frontend/src/models and enforce centralized typing, and move page-specific components under components/<feature>/ to align with AGENTS.md, or update AGENTS.md to permit route-level co-location of components and local types where appropriate. Given Next.js App Router norms, updating AGENTS.md to allow co-location is likely more pragmatic.

7) CI/CD folder naming
   - AGENTS.md mentions pipelines/ but not azure-pipelines/.
   - Actual:
     - Both pipelines/ and azure-pipelines/ exist with templates/variables.
   - Conclusion: Acknowledge azure-pipelines/ in the doc or clarify where pipeline definitions live to avoid drift.

8) Database source toggles
   - AGENTS.md: Only “Server: SQL Server”.
   - Actual:
     - Startup supports multiple data sources and explicitly enforces “non-Local cannot use Access.” DataExtensions reads Data:* settings, allows “Access” in Local, and otherwise throws if Access is selected.
   - Conclusion: Update the Database section to reflect the environment policy (Local may use Access repositories, non-Local must use SQL).

9) API error/response shapes and policies
   - AGENTS.md defines simple response shapes; code uses ProblemDetails in controllers for errors and tracks telemetry.
   - Actual:
     - Example in CovenantsController: returns ProblemDetails with structured extensions and logs exceptions.
   - Conclusion: Consider updating doc to bless ProblemDetails for errors and the policy-based authorization approach already in use.

10) Output order and DTO placement
   - AGENTS.md mandates DTOs under Casrr.Application/DTOs/<Feature>/… and a strict AI file delivery order.
   - Actual:
     - Application has Reviews/Contracts and Services, not DTOs in the specified DTOs/ folder. Interfaces are at the project root.
   - Conclusion: Either realign Application layer to the DTOs folder structure or update AGENTS.md to reflect the current layout (Contracts/Services with interfaces at root).

Concrete improvements to AGENTS.md

All controllers require [Authorize] with appropriate policy.” Document the diagnostics endpoints and Swagger security schemes already present.

    - Option B (update document, pragmatic): State that repositories use raw SQL via SqlClient and may use Dapper for mapping/perf; EF Core DbContext is available for selected scenarios; no stored procedures unless approved.
  - Whichever you select, ensure consistency across Infrastructure and update naming/locations accordingly.

- Project inventory:
  - Replace “7 projects” with an explicit list of current projects:

- Error handling:

- Naming and examples:
  - Update examples for service naming (drop “Service” suffix to match current code), and route examples to include versioning.

Bottom line

AGENTS.md is directionally useful but out of sync on several critical points (auth model, ORM/repository pattern, project count, routing/versioning, and frontend file-location/type placement). It should be revised to reflect the repository’s current implementations and conventions. Alternatively, if the AGENTS.md is the intended target architecture, then several areas of the codebase should be refactored to conform (auth from JWT to Windows/AD; data access from ADO.NET/Dapper to EF Core; relocate Application/Interfaces; move types to frontend/src/models; adjust component placement). Choose one path and make the doc/code consistent to avoid developer confusion and drift.
