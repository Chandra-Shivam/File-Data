# Task: Store archived blobs in a Year/Month/Day folder structure, auto-creating the container

## Context
This is a C# .NET Azure Functions (isolated worker) project that is blob-triggered: an
Excel file lands in the `originate-landing` container, is processed through a pipeline, and
the source file is then archived into the `originate-processed` container.

Storage access uses **managed identity** (`DefaultAzureCredential`) via the `OrigenateStorage`
connection — NOT connection strings, NOT `AzureWebJobsStorage`. Do not introduce connection
strings or `BlobContainerClient` constructed from them.

## Goal
When a blob is archived (i.e. when the file is "created" in the destination container),
it must be placed inside a date-partitioned folder path:

```
<Year>/<MonthFullName>/<DayTwoDigit>/<originalFileName>
```

Example for 2026-05-05:  `2026/May/05/MyFile.xlsx`

Notes on the exact format (follow precisely):
- Year: numeric 4-digit (`2026`)
- Month: **full month name** (`May`) — NOT the numeric `05`. Use invariant culture.
- Day: zero-padded 2-digit (`05`)
- Use **UTC** for the date (consistent with the rest of the codebase).

If the destination container does not exist, create it automatically before writing.

## Existing code to reuse / match (do not reinvent)
- `Infrastructure/Storage/BlobArchiver.cs` — `ArchiveAsync(string sourceBlobName, CancellationToken ct)`.
  This currently copies the blob to `_containers.Archive()` using a flat name and the
  `StartCopyFromUriAsync` + `WaitForCompletionAsync` + `DeleteIfExistsAsync(source)` pattern.
  **Modify this** so the destination blob name includes the Year/Month/Day prefix above.
- `Infrastructure/Storage/ContainerClientFactory.cs` — resolves container clients via
  `Landing()`, `Archive()`, `Failed()` from `OrigenateOptions`. Use `Archive()` as the destination.
- Container-ensure idiom already used in `Infrastructure/DeadLetter/ErrorArchiveBlobWriter.cs`
  and `Infrastructure/DeadLetter/StorageProvisioner.cs`:
  `await containerClient.CreateIfNotExistsAsync(cancellationToken: ct);`
  Use the same thread-safe one-shot guard (`SemaphoreSlim` + a `_containerEnsured` flag)
  as `ErrorArchiveBlobWriter` if you want to avoid calling it on every blob, OR just call
  `CreateIfNotExistsAsync` (it is idempotent) — match `ErrorArchiveBlobWriter`'s style.
- Date-partition formatting idiom already in `ErrorArchiveBlobWriter.cs`:
  `$"errors/{now:yyyy}/{now:MM}/{now:dd}/..."` — adapt it to use the full month name (`MMMM`).

## Requirements
1. Build the destination blob name as:
   `$"{now:yyyy}/{now:MMMM}/{now:dd}/{sourceBlobName}"` using
   `DateTimeOffset.UtcNow` and `CultureInfo.InvariantCulture` for the month name.
2. Before copying, ensure the destination container exists via `CreateIfNotExistsAsync`.
3. Preserve the existing copy semantics: server-side copy from the landing blob URI,
   wait for completion, then delete the source blob (move, not duplicate).
4. Keep the existing duplicate-name collision handling, but apply it WITHIN the dated
   folder: if a blob with the same dated path already exists, append a
   `-yyyyMMddHHmmss` timestamp to the file stem before the extension
   (mirror the current logic in `BlobArchiver.cs`).
5. Do not change the trigger, the pipeline ordering, or the auth model.
6. Match the project's logging style (`_log.LogInformation("EventName=... Blob={Blob}", ...)`).

## Acceptance criteria
- A file processed on 2026-05-05 named `Foo.xlsx` ends up at
  `originate-processed/2026/May/05/Foo.xlsx` and is removed from `originate-landing`.
- If `originate-processed` did not exist, it is created automatically.
- Month appears as the full English name (`January`…`December`), not a number.
- No connection strings introduced; managed identity path unchanged.
- Existing unit/integration tests still pass; add a test for the dated path + month-name
  formatting if a test project exists.
